图谱建成之后:维护与使用的工程设计

以沪铜为例,回答两个问题:① 事件如何自动化载入、图谱如何被人和 AI 共同维护;② 交易员/研究员/AI agent 如何用它发现盲点、完成推理。核心立场:不发明新机制——事件走管道,知识改动走 PR,git 就是人机协作的审批系统。

0. 协作闭环总览(四角色 × 三条流)

三条流:事件流(新闻/数据 → 事件日志,天级)、知识流(观点/发现 → PR → 图谱,周级)、推理流(图谱+事件 → 晨报/问答/仪表盘,实时)。人只出现在两个卡口:事件晨审(5 分钟/天)和 PR review——其余全自动。

1. 事件自动化载入:两条流水线,定性到定量

关键工程决策:数据型事件和冲击型事件分开处理——前者根本不需要 LLM,是"定性到定量"最可靠的第一步。

🅰 数据型事件流水线(全自动,先做)🅱 冲击型事件流水线(LLM+人审)
覆盖定期数据发布:SMM 社库(周一/四)、LME 库存(日)、现货 TC(周五)、精废价差(日)、MPOB(每月10日)、PMI、竣工、NEV…——约七成事件突发新闻:矿山事故、政策公告、罢工、关税、收抛储…
处理数据 API → 计算预期差 z-score或同期偏离(vs 5 年区间)→ 超阈值自动生成事件,magnitude 天然是定量的(z=2.3 而非"强")爬虫按 23 个根因的 signals 关键词包粗过滤(便宜高召回)→ LLM 解析:受控词表(只能映射到已有根因,防幻觉)输出 {time, root_cause, direction, magnitude 估计, window, 原文引用, confidence}
z-score
是什么		预期差 z-score =(实际公布值 − 市场预期中值)÷ 历史预期误差的标准差。例:MPOB 库存实际 226 万吨,路透调查中值 220 万吨,偏差 +6 万吨;历史上"实际 vs 调查偏差"的标准差约 4 万吨 → z = +1.5,读作"超预期约 1.5 个标准差"。两个作用:① 把不同指标的"意外程度"变成可比的统一刻度(库存的 6 万吨和 TC 的 5 美元没法直接比,z 可以);② 事件幅度从人工拍的"强/中/弱"变成客观数字——"定性到定量"在事件层的具体落点
入库高置信直接写 events.yaml(标记 auto,可回溯)一律进待审队列,研究员晨审确认(预计 5 分钟/天:去重聚合后每天只剩几条)
定量化z-score 即幅度;时间戳=发布时刻(point-in-time)LLM 估计物理量(影响产能占全球 %)→ 人工确认;多家媒体报道自动聚合为一条,报道数=热度信号
失败出口数据缺失 → 管道告警(工程师)归属失败 → 未归属事件队列 = 新根因候选的发现机制;反复出现的未归属事件自动提议新根因(走 PR)
💡 "定性到定量"的完整含义:图谱给方向和结构(定性);事件带z-score 幅度 + 指标节点挂当前读数 + 边上 condition 用读数自动判真伪(定量)。三者齐了,推理输出就从"利多"变成"利多,幅度参考历史同级事件均值 +3.2%,前提条件已满足 2/3"。

1.5 系统完整架构:一切都是仓库里的文件(回应"PR 具体长什么样")

先澄清一个误会:events.yaml 是一个文件装多条事件(append-only 列表,每条事件是文件里的一条记录),不是每个事件一个文件;事件量大之后按月分片(events/copper/2026-07.yaml)。完整仓库布局:

fundamental_research/                ← git 仓库(整套系统的全部状态)
├── kg-mvp/
│   ├── data/                        ★ 唯一需要人审的层:全是 YAML 文本
│   │   ├── copper.yaml              ← 图谱本体:节点(含 temporal/theme)+ 边(五要素+rationale)
│   │   ├── copper_events.yaml       ← 事件日志:一个文件多条事件,只增不改
│   │   ├── palm_oil.yaml / palm_events.yaml
│   │   └── (未来) events/copper/2026-07.yaml    ← 按月分片
│   ├── build_graph.py / propagate.py             ← 引擎(品种无关,很少改)
│   ├── gen_panel / gen_explore / gen_layers ...   ← 生成器
│   └── out/                         ← 构建产物(SVG/JSON/HTML片段),可随时重建,不入审
├── .lavish/copper-mvp.html          ← dashboard(构建产物)
└── .github/workflows/ci.yml         ← CI 质量门(schema/golden tests/保质期扫描)

数据流: data/*.yaml ──构建──▶ out/* ──注入──▶ dashboard
审批流: 只审 data/ 层的 diff;out/ 和 dashboard 跟着自动重建

PR = 对 data/ 层的一次文本修改。两个具体样例:

例 A · Agent 起草的「新增边」PR(L2,需研究员批)

PR #42: 新增边「废铜供给→精废价差」
来源: XX研报 2026-06-15 · 起草: agent

--- a/kg-mvp/data/copper.yaml
+++ b/kg-mvp/data/copper.yaml
@@ 废铜替代段 @@
+  - {from: 废铜供给可得性, to: 精废价差,
+     polarity: "-", lag: 1-2月, strength: 弱,
+     rationale: "废料紧张时持货商挺价,
+       精废价差被动收窄...",
+     evidence: "XX研报2026-06-15:东南亚
+       拆解产能外移致国内废料紧张"}

CI: ✅schema ✅golden ✅环检测(无新环)
研究员 Approve → merge → dashboard 自动重建

例 B · 每日事件入库 commit(数据型自动;冲击型晨审后)

commit: events 2026-07-06 (auto + 晨审)

--- a/kg-mvp/data/copper_events.yaml
+++ b/kg-mvp/data/copper_events.yaml
@@ 文件末尾追加 @@
+  - {time: "2026-07-06", root_cause: 中国社会库存,

+     source: "SMM周一社库31.2万吨,低于5年同期P10"}

+     direction: +1, magnitude: 中, window: 待反转,

研究员看到的永远是这种 diff:一行一个事实、带来源、CI 已把关。批一个 PR ≈ 审一条消息的工作量,不是读代码。⚠️ 前置事项:这套机制以 git 为地基,fundamental_research 目前尚未 git init——启用 PR 工作流前需先补上。

2. 图谱本体维护:git-native,人机同一通道

一切修改 = PR,人改和 AI 改走同一条路。"AI 修改需人类同意"不需要发明审批系统——PR review 就是审批系统,diff 可读(YAML 一行一个事实)、可回滚、有历史。

变更风险分级(决定谁能批)

级别变更类型审批
L1 低风险补充 evidence/案例、增加 signals 关键词、theme 状态更新(有事件密度数据支持时)CI 通过即可自动合并(或事后抽查)
L2 中风险边强度/时滞调整、condition 修改、新增边、事件窗口默认值单研究员 review
L3 高风险新增/归档根因、polarity 翻转、机制重写、schema 变更双人审或周会过

CI 质量门(工程师维护,每个 PR 自动跑)

  • schema 校验(已有:悬空节点/非法枚举/重复边)+ 反馈环变化报告(新增或消失的环要在 PR 里显式确认)
  • 推理回归测试(golden tests):一组断言,如"矿端扰动+1 ⇒ SHFE 利多""出口限制+1 ⇒ 必须检出政策自反环"——防止一条边改错让整图输出漂移。「政策自反环」澄清:不是"一个新闻推出两个矛盾结论",而是同一冲击沿不同时滞的路径产生先后相反的阶段效应——禁令当下利多(供给收紧),但同时启动"国内胀库→倒逼政策放开"的回路,数月后转向利空(2022 印尼 25 天解禁的真实剧本)。这不是矛盾,是分时段结论的价值所在。golden test 断言的是:无论边表怎么改,推理器必须仍能检出这个回路——保护这类结构性知识不被误删
  • 边保质期检查:review.last 超过 N 月的边列清单提醒(工程细节见下面「时间系统的工程落点」)
  • 跨品种一致性:共享根因(原油、美元、美联储)在铜/棕榈油图里的机制方向必须一致

时间系统的工程落点(回应"三层/保质期怎么作为工程放进来")

之前设计的时间系统(回归特性/事件窗口/主题热度/保质期)在工程上就是四个字段 + 三个定时任务 + 一个过滤函数,全部落在已有文件里:

时间概念落在哪个字段由什么程序消费
根因回归特性copper.yaml 节点的 temporal.reversion(已落地)事件缺省窗口查表;推理分桶展示
事件影响窗口events.yaml 的 window 字段(已落地)核心是一个过滤函数:活跃事件 = (window=待反转 且日志中无后续反向事件) or (time+window ≥ 今天)——目前 active 是手工标的,自动化就是实现这 3 行判断,跑在每次推理之前
主题热度节点的 theme.status每周 cron:统计各根因 90 天事件密度 → 建议升/降档 → 生成 L1 级 PR(人可秒批)
边保质期边的 review: {last: "2026-07"}(待补字段)每月 cron:扫描 today − review.last > 12月 的边 → 清单 issue @研究员;确认无误只更新日期(L1),要改就走 L2

没有任何新系统:字段在 YAML 里、任务是三个 cron 脚本、判断是纯函数——时间系统 = 数据模式 + 定时器。

维护节律(谁在什么时候做什么)

节律动作角色耗时
每日事件管道自动运行;晨审待审队列;agent 生成晨报Agent + 研究员5 分钟
每周主题热度按事件密度刷新;冲突仪表盘/面板重建;PR 批处理Agent + 研究员30 分钟
每月超期边巡检(CI 清单);未归属事件队列复盘(要不要新根因)研究员1 小时
每季/大行情后图谱验尸:对照事件日志复盘——这轮行情图谱是否提前给出结构?哪条边失灵?→ 回测校准(甲醇报告方法),validation 字段更新研究员 + 工程师半天

除了"人审 + AI 提 PR",另外四种可行的维护机制

① 行情驱动的异常检测(最推荐)
价格大幅波动但图谱无活跃事件可解释 → 自动开 issue"疑似缺根因/缺事件";反向:图上事件活跃但价格不动 → 主题计价死亡信号。让市场本身当图谱的质检员。

实现(一个收盘后 cron 脚本,约百行,无需实时系统):
① 触发:|当日收益| > 2.5×(60 日滚动σ),或 3 日累计超阈值;
② 解释度检查:查事件日志——近 3 天是否有活跃事件,且该事件经图传导到 SHFE 的方向与价格变动一致、强度不弱;
③ 三分类输出:
· 有一致事件 → 正常,把"事件-价格反应"记入配对库(顺便积累 event study 数据,为 price-in 检查铺路);
· 无事件 → issue「价格异动无解释」,附当日新闻关键词粗过滤 top 命中,人 5 分钟内判断是漏事件还是缺根因还是纯资金;
· 有事件但方向相反 → issue「图谱方向失灵」,涉事边标记待校准;
④ 反向检测:事件 z-score 大但 |价格反应| < 0.5σ 连续 N 次 → 建议该主题降档(计价死亡)。
② 研报批量对照
agent 定期读卖方研报,diff "研报提到的机制 vs 图上已有的边" → 缺失项生成候选边 PR(带研报引用)。比读新闻更高效:研报本来就是结构化观点。
③ 回测校准循环
甲醇报告的方法:每季对带 factor_rule 的边跑相关性检验+阈值遍历,validation 自动更新;胜率持续退化的边自动降档并@研究员。强度从拍脑袋变成有保质期的实证值。
④ 批注即贡献(已在用)
交互探索页点边批注 → agent 转化为 PR。研究员不需要会 git,点图说话就是维护;lavish 的批注流程就是这个机制的原型。

3. 使用:三种人,三种时间尺度,各取所需

👤 交易员(日内-周):四件套

晨报(自动)隔夜新事件 → 传导子图 + 分时段结论 + 今日待验证信号。回答"今天为什么会动、动了先看什么"
事件速查盘中突发新闻 → 30 秒内:传导链 + 历史同类事件对照(事件日志就是案例库:"2022.4 禁令当晚 +6.9%,25 天后反转")
冲突仪表盘定义:一张自动刷新的清单,列出图上当前"多空路径同时被活跃事件激活"的节点。例:当下 SHFE 铜的 6 月+ 时段,"精铜供给"同时挂着多头路径(CSPT 减产)和空头路径(硫酸养冶炼→减产不兑现)——它就会出现在仪表盘上,旁边附裁决信号(下一期 SMM 精炼铜产量数据能定方向)。这就是"今天行情的分歧点在哪、看什么数据能解决"。借鉴甲醇报告实证:冲突期不开仓,等裁决
Price-in 检查事件发生 N 天、价格已走 X%,对照历史同级事件的平均反应幅度 → 估计剩余空间(需要 event study 数据积累后上线)

👤 研究员(周-季):图谱是假设的显性化工具

写观点前跑反向归因(explain_drivers)对照自己的逻辑清单——"我漏了哪条路径?"尤其看异号路径提示
写观点时把观点翻译成"我认为路径 X 主导"——图谱自动列出该路径的全部 condition:观点成立需要什么前提、什么信号会证伪它。观点从散文变成可检验的结构
写完观点观点里的新机制/新条件 → 边表 PR,研究成果直接沉淀为资产(而不是躺在报告里)
复盘行情结束对照事件日志:哪条路径兑现、强度对不对 → validation 更新。研究质量第一次变得可度量
带新人交互探索页 = 品种知识的自学教材(点边看机制、rationale、案例)

🤖 AI Agent(持续):三端角色,中间永远不裸推

  • 输入端:事件解析(受控词表)、数据型事件生成、研报对照、候选边起草——图谱的"感知器官"
  • 推理端:只通过 MCP tools 调 propagate/panel/explain,传导永远由符号引擎执行——既定架构不变
  • 输出端:晨报生成;自然语言问答("Grasberg 复产对沪铜什么影响?")回答必附传导子图与依据链;魔鬼代言人:对任何人类观点自动生成反方路径——即图上与你观点方向相反的传导路径。例:你说"供给缺口看多铜",agent 自动列出图上现存的空头路径("硫酸价格↑→冶炼利润↑→减产不兑现→精铜供给不减")并附其前提条件与验证信号。不是 agent 自己反驳你,是把图上已有的反向机制摆到你面前——盲点发现的日常化

🔧 工程师

管道 SLA(事件流水线的监控告警)、CI 质量门、MCP tools、数据接入(point-in-time 登记每个指标的发布时点)、回测框架。工程师不需要懂商品——所有领域判断都在 YAML 和 review 里,这是 KG-as-code 分工的本意。

4. 专节:盲点是怎么被这套系统发现的(四个机制)

盲点类型发现机制例子
① 我不知道这个机制存在
(覆盖盲区)		未归属事件队列:反复出现却挂不到任何根因的新闻 = 市场在发生、图谱没有的机制;+ 研报对照 diff假如"再生铜拆解产能出海东南亚"的新闻反复归属失败 → 提示需要新根因
② 机制我知道,但我没在看
(注意力盲区,最值钱)		休眠区巡检:定期扫描休眠主题的根因,其数据读数已在极端分位却无人交易 = 机制在积累势能、市场和你都没看——图谱记得你忘了的东西地产竣工休眠两年后,若竣工数据连续回升至 5 年区间上沿而盘面无反应 → 巡检报告置顶提示
③ 我只想到了一边
(路径盲区)		反向归因 + 异号路径提示:"你的利多观点,图上还有 3 条利空路径,最强的一条经硫酸→冶炼利润";agent 魔鬼代言人把这变成每篇观点的默认流程2026 供给缺口叙事下,图谱始终亮着"硫酸养冶炼→减产不兑现"的反方路径
④ 我以为它还有效
(时效盲区)		边保质期 + 回测校准:胜率退化的边自动降档;theme 计价死亡检测(事件继续、价格不响应)"竣工-20% 该跌"的 2021 直觉,被 theme=休眠 与敏感度衰减数据拦住
