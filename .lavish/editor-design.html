UI 编辑器 → PR:让非技术同事也能维护图谱

目标:同事只填表单、不碰 YAML/git;系统把表单变成一个规范的 PR,CI 自动守门,懂行的人点一下 merge。

三条原则:① 编辑器是表单(下拉/按钮/文本框),不是代码编辑器,没人会看到 YAML 语法; ② 提交产出的是规范 PR(带作者、来源、可 diff、可回滚)——沿用我们定的"PR review = 审批系统"; ③ 提交前先跑一遍 CI(schema + golden tests),非技术用户不可能提交出坏图。

1. 能编辑什么(按频率/风险分三档,分批做)

档操作风险技术难点次序
加事件加一条新闻/数据事件(选根因+方向+幅度+来源)低(append-only,不改机制)无——直接追加一行先做
改边/改主题调边的强度/rationale/condition;改主题 活跃⇄降温⇄休眠中(改机制强弱)YAML 改现有条目需保留注释(用 ruamel)次做
加/删 根因或边新增根因节点、连新边高(改结构)结构校验、跨品种一致性最后

建议第一版只做「加事件」:它是日常最高频、最安全的操作,正好是"看到新闻 → 录入"的核心工作流,非技术同事天天要做的就是它。改边/加节点随后跟上。

3. 【关键决策】PR 到底怎么被创建出来?

A. 后端本地 git + gh(推荐先用)B. GitHub App/TokenC. 变更队列后转 PR
怎么做后端在一个独立 worktree 里建分支→写改动→commit→push→gh pr create后端用 GitHub API + 服务器上的 token 直接开 PR提交先写入待转队列,人工/定时器再批量转 PR
现在能用吗能,零额外配置——本机 gh 已登录(RioAraki)要先建 GitHub App / 细粒度 token能,但多一道人工
上服务器服务器要配 git 凭据生产级最佳(token 存密钥管理)队列可跨环境
并发安全用独立 worktree,不碰运行中的工作区API 层无本地状态,最干净天然串行

推荐:代码里抽一个 create_pr(分支, 改动, 标题, 正文) 函数,第一版用 A(本地 gh,现在就能跑),将来上服务器把实现换成 B(token),上层表单代码不用动。这样不被基础设施卡住,又给生产留了路。

4. 提交前校验 + 谁审批

  • 提交前即时校验:后端在临时副本上跑 ci_checks(schema + golden tests + 事件合法性)。加事件极少会失败(顶多根因名不存在,但表单是下拉不可能选错);改边可能触发 golden 失败 → 当场提示"这个改动会让'矿端扰动→利多'的断言失效,请确认",挡在 PR 之前。
  • PR 作者:表单里记提交人,写进 commit 的 author 与 PR body,谁提的一目了然。
  • 审批 = merge:沿用既定的变更分级——加事件/改主题状态(低)可快批甚至自动;改边强度(中)单人审;加删节点(高)双人审。merge 后服务重启/热重载即生效。
  • 非技术友好的审批界面:审批人也可以不看 GitHub,在站内看"人话摘要 + 校验结果"点通过(站内审批调用同一个 merge)。
