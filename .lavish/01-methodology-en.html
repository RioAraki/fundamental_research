Commodity Fundamentals Root-Cause Knowledge Graph — Methodology

Turn the analyst's supply-demand transmission logic into a knowledge graph that is reasoned over, maintainable, and explainable. This page covers only the "why" (methodology); the "how to implement in code" lives in the companion page Engineering.

In one sentence: we do NOT chase "end-to-end AI price prediction". Instead we make the analyst's causal framework explicit as a root-cause graph with direction / lag / strength; a symbolic engine deterministically computes the full transmission chain, then hands the entire chain to an LLM to interpret — the LLM reads the entrance (parses events) and explains the exit, seeing the whole reasoning process but never doing the propagation itself. The graph is an asset, the reasoning is explainable, and the knowledge is collaboratively maintainable.

0. Methodology at a glance (one diagram)

Each layer has its job: the root-cause layer (slow, the graph itself) provides transmission structure; the event layer (fast) activates it; the theme layer (medium) tunes pricing weight. The symbolic engine computes a full derivation chain along the root-cause layer; that chain is handed to the LLM as a prompt, yielding conclusions that are time-bucketed, explainable, and that do not force-adjudicate conflicts.

1. Seven core principles (not to be casually overturned)

#PrincipleWhy
1KG-as-code: the single source of truth is YAML text in git; the graph DB / pages are all regenerable artifactsdiff / rollback / blame / PR review; same collaboration flow as analysts writing docs
2Neutral root causes: a root cause is a neutral state variable (e.g. "India import tariff"), its name contains no direction word; a direction field explains the +/− semanticsthe same root cause moves both ways (tariff up / down); baking in a direction would miss half the moves
3Graph stores mechanism, news sets direction: an edge's polarity is the transmission mechanism (source↑ → what happens to target); events enter as ±1 shocksmechanism is stable (expert-maintained), direction is time-varying (news-driven) — keep them separate
4Three-layer ontology: root cause / theme / event (next section)only by separating "mechanism, attention, observation" — three different time scales — does it become coherent
5Do not adjudicate conflicts: output bullish vs bearish paths side by side; the action semantics = stand aside during conflict + give the resolving signalthe conflict itself is a research signal; forcing a single direction throws away information
6Engine computes transmission, the full chain is fed to the LLM: propagation is executed deterministically by the symbolic engine (the LLM does NOT reason it out — avoiding hallucinated paths); but the computed full derivation chain (each step's direction/lag/strength/condition) is passed to the LLM as part of the prompt, so the LLM — while reading the entrance (parsing events) and explaining the exit — sees and can cite the entire reasoning chainan LLM would confuse correlation for causation and hallucinate paths, so it must not do the propagation; but seeing the full chain lets it give a solid, traceable interpretation rather than guessing from the two ends only
7Humans edit directly, AI edits need human review: every change is a PR; a human may edit and submit directly, while any change drafted by AI (from reports/news) must be reviewed by a human before it merges — PR review is the approval gateno need to invent an approval system; git already has diff / rollback / history / CI gating; AI only drafts, humans decide — quality and accountability stay clear

2. Three-layer ontology: root cause / theme / event

The one-line test: strip the time and direction out of a news sentence; the noun that remains is the root cause.

LayerDefinitionTime scaleStored inExamples
🧱 Root causea measurable, persistent, neutral state variable; a graph node (incl. intermediate transmission nodes)years-decadesYAML graphmine-supply disruption rate, India import tariff, property completion activity
⚡ Eventone timestamped ± assignment to a root cause (from news/data)hours-weeksYAML event log"2025.9 Grasberg accident" = mine disruption +1
🔥 Themethe market-attention state of one/several root causes over a period; a tag attached to the nodemonths-yearsa field on the node"AI compute build-out", "property-chain collapse (dormant)"
Root-cause reversion type (5 kinds): how it returns to baseline after a shock — fast-reset (monthly-report surprise) / event-recovered (a strike) / state-held (an export ban, "until reversed") / trend-accumulating (tree age, never returns) / seasonal. This determines the end of an event's impact window.
Why the theme is its own layer: the same "completions −20%" caused a big drop in 2021 (theme active) but no reaction in 2025 (dormant) — the mechanism didn't change, what changed is whether the market is looking. Theme = a pricing-weight multiplier; dormant means discounted, not deleted.

Confusing examples (the best way to tell the three layers apart)

In everyday speech you don't separate them, but in the system you must place each on the right layer. These are the easy-to-confuse ones:

You'll hearIt is actuallyWhy it's confusing / correct decomposition
"Inventory rose again"⚡ Event"inventory" itself is a root cause (node); "it rose by X kt this week" is the event; "the restocking cycle" is a theme — the same word appears on all three layers
"The Fed hiked"⚡ Eventthe root cause is "Fed policy stance" (there's always a current stance); "this hike" is just one +1 assignment to it
"Grasberg had an accident"⚡ Eventit sounds like a "cause", but it's one timestamped observation; the root cause is "mine-supply disruption rate" (a persistent state variable with a value at any moment)
"Indonesia banned exports"⚡ Event (+1)the root cause is "Indonesia export-restriction intensity" (neutral, can tighten or loosen); the ban is the +1 event, the later flush-out is a separate −1 event
"AI data-center demand"🔥 Theme (not a root cause)the name looks like a root cause but stuffs a narrative into the naming. The root cause should be the neutral "data-center / compute infrastructure demand" (it existed in 2015 and will exist after AI cools); "AI compute" is a theme tag on it that comes and goes
"It's a supply-deficit market now"🔥 Themeit sounds like a variable but maps to no single node — it's a story woven from several root causes (mine disruption + zero TC + grade decline); theme:root-cause = many-to-many
"Grid investment is strong"🧱 Root cause (a reading)this is the current level of the root cause "China grid investment" (a slow variable with a monthly reading); "the State Grid announced an annual plan +18%" is the event

Mnemonic: if you can ask "what is its value right now" → root cause; a sentence with a timestamp and direction → event; "the market is all trading X lately" → theme.

3. Reasoning: QPN symbolic propagation

On a signed (±) causal graph, propagate from the event node along paths: signs multiply (in series), strengths multiply (weak edges decay), lags accumulate; loops are handled by "no revisiting within a path + return-to-source recorded as a feedback loop". Each computed path is a derivation chain the LLM can cite.

Conflict handling: when bullish and bearish arrive at the same node → do not adjudicate; output the bullish paths vs the bearish paths + each side's strength; conflict = stand aside, await the resolving signal.
Lag → time buckets: bucket by cumulative lag into 0-1 mo / 1-6 mo / 6+ mo and sum each separately — answering structures like "bearish near-term but bullish long-term".
Explainable: every conclusion = a full transmission chain (with direction/lag/strength/condition); click a node to see the bull/bear verdict, click an edge to see "why this strength".

An edge has only two kinds: same-direction (+) = source up → target up; opposite (−) = source up → target down. Walking the chain, track whether each node actually goes up or down:

start  Drought ↑
  │ opposite(−): drought vs yield are opposite → drought up, yield DOWN
  ▼    Yield ↓
  │ same(+): yield and output move together → yield down, output DOWN too
  ▼    Output ↓
  │ same(+): output and inventory move together → output down, inventory DOWN too
  ▼    Inventory ↓
  │ opposite(−): inventory vs price are opposite → inventory down, price UP
  ▼    Price ↑        ⇒ Drought↑ ultimately drives Price↑ = bullish

Why does "some +, some −" end up +? — we are not adding the signs, we are multiplying them, "negative × negative = positive" style: a chain with an even number of opposite(−) edges is net same-direction; an odd number is net opposite. This chain has 2 opposite edges (drought→yield, inventory→price); the two "opposites" cancel → net same-direction, so "drought up" travels all the way to "price up". (−)×(+)×(+)×(−)=+ is just shorthand for "count the minus signs".

4. Root cause = qualitative, event = quantitative (quantifying is hard — don't pretend it's automatic)

The qualitative part (the graph) is relatively stable and can be accumulated: where a root cause transmits, through which intermediate nodes, in what direction — this is domain knowledge, expert-maintained. The hard part is quantifying — "how big is this move". Honestly, there are two cases with vastly different difficulty:

CaseHow to quantifyCoverage / difficulty
A few periodic data series that have both a number AND a market expectation (MPOB stocks, social inventory, TC…)compute a z-score = (actual − market expectation) ÷ std-dev of historical surprises; the magnitude is an objective numbercovers only a small subset; and it requires you to have that "market expectation" figure (Reuters/Bloomberg survey, or approximate with a historical mean) — which isn't always available
The vast majority of events (policy / shocks / strikes / report views)there is no clean numeric surprise; the magnitude can only be a manual strong/medium/weak judgment by the analystsubjective, experience-driven — the most human-dependent and hardest link in the whole method; we do not pretend it can be automated
  • Stated honestly: the z-score is a "compute-it-if-you-can" bonus (covering a few periodic series), not a general capability; quantifying event magnitude currently relies mainly on manual analyst input — this is a known hard bone, to be calibrated only gradually via historical backtesting, not solved in one step.
  • Each root cause's monitoring signals (news keywords + data sources) are a checklist of "what to look at, where to find it" that lowers the cost of the manual judgment — but the judgment itself is still made by a human.
  • The graph does not become useless just because it "can't give a precise number" — it first makes the direction and structure (qualitative) clear and leaves the magnitude for a human to fill; that alone is a big step up from "guessing causal chains off the top of one's head".

5. How the graph helps people find blind spots (the core value)

Human attention is limited, memory fades, and we tend to see only one side. The graph doesn't. Here is exactly how three mechanisms work, each with an example:

① Dormant-zone sweep — cures "I know the mechanism, but I'm not watching it" (attention blind spot, the most valuable)

How it works: a scheduled job (e.g. weekly) scans the root causes behind every theme with status=dormant → takes the current reading of that root cause's monitoring data → computes its percentile in the historical distribution → if the reading is already at an extreme percentile (e.g. >P90 / <P10) yet the root cause has had almost no recent events (news density ≈ 0) → it flags "the fundamentals are building potential energy while neither the market nor you is watching" → pins it to the top of the sweep report.

Example: the "property completion activity" theme went dormant after 2023. In some month of 2026, completion data keeps recovering to the top of its 5-year range (P92), but nobody is trading the property logic on the board and related news is near zero. The sweep surfaces it: "⚠️ property completions have reached a 5-year high yet the theme is dormant; mechanism: completions↑ → end copper demand↑, possibly underpriced by the market — recommend reviewing whether to reactivate the theme." — the graph remembers, for you, something you set down two years ago that may now matter.

② Counter-paths — cures "I only thought of one side" (path blind spot / devil's advocate)

How it works: the analyst states a view (e.g. "bullish SHFE copper") → the system propagates to the target price and picks out all paths pointing in the opposite direction to your view (those driven by currently-active events first, then structurally-existing ones) → each is annotated with the transmission chain, strength, lag, preconditions, and the signal that would verify it. The subtext: these are the things your view must, one by one, overcome to hold.

Example: you write "2026 supply deficit, bullish copper". The system auto-lists the counter-paths:
· [bear 0.6] sulfuric-acid price↑ → smelter margin↑ → production cuts don't materialize → refined supply doesn't shrink (precondition: by-product revenue stays high; verifying signal: SMM monthly refined-copper output);
· [bear 0.66] US tariff exemption → COMEX premium collapses → global inventory flows back → LME under pressure (verifying signal: COMEX-LME spread).
→ It forces you, before drawing a bullish conclusion, to answer: "why don't these two beat the deficit logic?" The view becomes testable rather than wishful.

③ Edge shelf-life + theme pricing-death detection — cures "I assumed it still works" (staleness blind spot)

How it works (two sub-mechanisms):
· Edge shelf-life: every edge carries review.last (the last time a human confirmed it); a monthly scan flags edges past due (e.g. >12 months un-reviewed) → reminds the analyst to recheck whether the direction/strength still holds.
· Theme pricing-death: for an active theme, if its events are still occurring (news density not low) but the price's sensitivity to those events approaches 0 (event study: on days a same-direction event appears, price no longer reacts) → judge "the bullish/bearish case is exhausted, fully priced in" → recommend downgrading the theme to dormant.

Example (shelf-life): the edge "property completions → end copper demand" deserved "strong" in the 2010s; as property's share of copper demand fell it is now "medium". If its review.last is stuck at 2024 and it's been >12 months un-reviewed by 2026 → the sweep reminds "this edge hasn't been reviewed in 2 years; property's share may have shifted again, please confirm the strength".
Example (pricing-death): in 2025 the "Section 232 tariff" theme initially made COMEX swing violently on every tariff headline; later, tariff news kept coming but price barely reacted → sensitivity decayed to near 0 → the system recommends downgrading the theme (the market has digested it), stop trading it as an active narrative.
