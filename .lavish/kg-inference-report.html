商品供需知识图谱 + 规则演绎推理

可行性与实现方式调研 —— 面向「基本面因素图谱化 → 喂给 AI agent 决策」

调研范围:金融/商品 KG 学术与业界实践(FinDKG、FinKario、哈工大事理图谱、数库 SAM 等)、推理引擎全谱系对比(Datalog/OWL/Prolog/产生式/定性推理)、KG×LLM agent 结合模式(GraphRAG/neuro-symbolic)、技术栈选型(2026 年 license 与维护现状)。全部出处见「参考文献」页。

总判断:可行,且你们的起点比大多数团队好。难点不在图数据库或推理算法(这个规模下都是解决了的问题),而在知识建模纪律——把研究员脑中的"利多利空传导逻辑"变成带方向、时滞、强度、条件的边表。你们已有的 markdown 知识库(产业链关系、利润公式、驱动因素)正是这张图的天然种子。

判断 ①:图要「专家骨架优先」

不要指望 LLM 从新闻自动长出可用的因果图——国内事理图谱路线试错后的共识是:纯自动抽取噪声大、粒度不齐、难落地交易。正确姿势:研究员人工定义每品种 50~200 条核心因果边,LLM 只做证据填充与候选发现。

判断 ②:推理用轻量符号传播

「检修→供给↓→价差↑→成本↑」这类推导,学术上是定性概率网络(QPN)的符号传播问题,有现成算法。几千节点+几百规则的规模,纯 Python(PyReason 或自写 200~500 行)就够,不需要重型语义网栈。

判断 ③:传导绝不让 LLM 裸推

2024-2025 金融 neuro-symbolic 文献的一致架构:LLM 管两端(事件理解、结果解读),符号引擎管中间(确定性传导)。LLM 单独做因果传导会把症状当原因、幻觉出图上不存在的边。

推荐总体架构:KG as Code

图数据库在这个规模下应视为可再生的构建产物,而非事实源。事实源放 git 里的文本(markdown + YAML 边表),CI 校验构建,运行时是 networkx 内存图 + 推理层,通过 MCP tools 暴露给 agent。零运维、全程 git diff/review、和现有知识库协作方式无缝衔接。

🧱 知识图谱怎么建

先说结论:商品供需因果没有现成权威本体,自己定义轻量 schema 是必经之路。但不用从零发明——FinDKG 的关系表(Positive/Negative Impact On)、CrudeOilNews 的商品事件类型表、ChainKnowledgeGraph 的产业链关系、加上你们自己的 markdown 知识库,拼装即可。

1. 四层分层 Schema(综合文献的可操作方案)

层放什么你们知识库里的对应物
L1 实体层品种、产业链环节、装置/产能、企业、地区、政策主体9 品种清单表、三条产业链 ASCII 图(已有,可直接转)
L2 指标层供需平衡表科目(产能/产量/开工率/进出口/库存/表观消费/利润加工费)作为 Indicator 节点;加工费等作为派生指标节点,公式存属性利润公式组:PTA加工费 = PTA − 0.66×PX 等(已有,可直接转)
L3 因素/事件层抽象因素(天气、检修、政策)与事件实例(某装置某日检修,带时间戳)分开建;因素→指标用因果边L4 驱动因素章节的文字(需显式化成边表)
L4 机制层"利润是供给的调节阀"这类研究员逻辑作为机制节点,把多条因果边组织成可解释的传导路径,供 agent 引用作 reasoning trace"核心因果链"文字段落(需显式化)

2. 边模型:利多利空 / 时滞 / 强度 / 条件 怎么表达

这是整个图谱的灵魂。因果 KG 文献的标准做法是带属性的因果边:

# YAML 边表示例(事实源,进 git)
- from: 装置检修(PX)
  to: PX进口量
  polarity: "-"          # 利多利空方向:same(+) / opposite(−)
  lag: "1-3周"           # 传导时滞
  strength: 0.7          # 强度 [0,1],FCM 式权重,先拍脑袋后校准
  condition: "检修产能占比>5%"   # regime 依赖,推理时过滤
  mechanism: 供给收缩     # 挂到 L4 机制节点
  evidence:              # 强制溯源,否则无法审计
    - "2023-03 S-OIL 检修案例,PXN 走扩 $40"
    - "01-品种/化工板块-总览与产业链.md#L4"
方向(polarity):FinDKG 用 Positive/Negative Impact On 关系类型,是最简可用的利多利空建模。等价于模糊认知图(FCM)的 ± 权重——商品供需框架本质上就是研究员脑中的 FCM。
时滞(lag):边上标滞后期数;推理时携带累计时滞,可参考时序因果发现的 window causal graph(只保留最大滞后窗内的边)。
条件(condition / regime):没有标准本体,三种通行做法——① 边上挂 condition 属性(推荐起步);② 把 regime 具象化为节点,用 RDF-star 式 qualifier 指向"边本身"(如"高库存状态 −弱化→ 这条边");③ 每个 regime 一张子图层。
观点 vs 机制:研报里"我们认为 X 利多 Y"是观点,应作为带出处/分析师/时间的 assertion 建模,不要与验证过的机制混存——这是事理图谱落地批评的核心教训。

3. 产业链与平衡表怎么挂图

  • 产品流:Material 节点 + 有向边,边上挂转换系数——PX -consumes{ratio:0.66}-> PTA。这使 agent 能做成本传导计算(PX 价格→PTA 现金成本→加工费)。你们的"配方记忆"(聚酯 ≈ 0.86×PTA + 0.34×EG)就是现成的系数表。
  • 量不进图:Indicator 节点只存元数据(名称、口径、频率、单位、数据源),真实时间序列放外部时序库(TimescaleDB/ClickHouse/甚至 parquet),节点用 ID 关联。agent 查询时:图上走因果路径 → 落到指标 → 去时序库取数计算。
  • 恒等式进图:供应=产量+进口−出口、库存变化=供应−消费,作为图上的约束/派生规则显式建模——agent 可用它做一致性检查和缺口推算。
  • 装置级建模:每环节挂 Plant 节点(产能、工艺、投产时间、检修状态),事件指向 Plant,Plant 汇总到产能指标。检修/投产这类"化工特有的高频供给冲击"就有了挂点。

4. 构建流程:自顶向下 vs 自底向上

文献共识是混合、但 schema-first。FinKario(研报→KG→回测胜机构 17.85%)、FinDKG、供应链风险监控全部是先定 schema 再 LLM 填充。纯自底向上(无 schema 抽取)产出碎片化子图——哈工大事理图谱(134 万事件节点)最终被业内批评"利多利空混淆条件、粒度不齐、难落地交易"。

专家从 markdown 提炼 schema
人工建每品种 50-200 条黄金边
LLM 抽事件实例+候选边
候选进待审队列,人审合并
数字一律从平衡表数据库回填
LLM 抽取环节当前准确率(2024-26 文献)对策

5. 已知的坑(文献归纳,提前避)

① 浅层覆盖:LLM 只抽显式表述,漏隐含逻辑链,图推理连不起来 → 骨架必须人工建
② 粒度漂移:同一因素被抽成 N 种表述,消歧归一是工程量最大的环节 → 受控词表(枚举合法实体/关系类型写进 prompt)
③ 无溯源:边上强制 evidence(原文句+来源+日期),否则无法审计
④ 时效:静态 KG 反映不了装置投产/检修动态 → 事件带时间戳;过期的图比没有图更危险
⑤ 因果≠相关≠观点:三者分开建模,观点带作者与时间
⑥ 数字幻觉:LLM 抽的数一律不信,回填校验

⚙️ 推理怎么做(rule-based deductive inference)

先定位问题:你要的推理,学术上叫「定性概率网络(QPN)上的符号传播(sign propagation)」——在带 ± 号的因果图上,从证据节点出发沿路径传播影响方向,符号相乘(传递)、相加(汇聚)。有 AAAI 1993 起的成熟算法可直接照抄思路。「装置检修→供给↓→价差↑→成本↑」就是一次前向传播。

🤖 怎么喂给 AI Agent

三层分工:2024-25 金融 neuro-symbolic 文献的一致架构

FinRipple(事件涟漪效应预测)、SHIELD(供应链中断预测)、LLM 生成贝叶斯网络做期权决策——pipeline 模式高度一致:

为什么中间不能让 LLM 裸推:可观测性领域的结论是 LLM"把症状当原因"、幻觉根因;传导必须由图算法/符号规则确定性执行,LLM 只负责两端。这与你们 px-agent.html 蓝图里"[事件]→[方向]→[量级]→[时滞]→[price-in]→[置信度]"的输出模板天然吻合——那个模板的中间四项正是符号引擎的输出。

GraphRAG:借思想,不走它的路

不需要它的建图管线:GraphRAG 是为"从海量文本自动长出图"设计的(LLM 抽取+社区检测+逐级摘要),抽取准确率 60-85%、实体消歧是最常见崩溃点、大语料 ingestion 成本 $85k-175k。你们的图是人工构建、实体可控,直接跳过 indexing,只用检索侧思想。
实证边界:简单事实检索上朴素 RAG 反超 GraphRAG(60.9% vs 49.3%);多跳推理才反转;MS GraphRAG global search 单查询 ~33 万 token vs 朴素 RAG 879 token。
最值得借鉴的是 PathRAG:在查询相关节点之间检索关键关系路径并剪枝(而非整个邻域/社区)——路径本身即推理证据,与"事件 X 沿产业链怎么传导"的问法一比一贴合。

Tool call 查图 vs 子图塞 prompt:组合用

方式实证现状用在哪

推理结果的呈现格式:路径即证据

  • 输出完整传导链而非结论摘要:装置检修 →(−供给) PX供应 →(−) PX价格↑ →(+成本) PTA成本 →(…)。RoG(ICLR'24)、Paths-over-Graph(WWW'25)证明:高相关知识路径显著提升 LLM 忠实度;Graph-constrained Reasoning 更激进——解码阶段用 KG-Trie 约束 LLM 只能说图上存在的路径,杜绝幻觉边。
  • 每条结论回链到图中证据(graph-backed traceability):边上的 evidence、机制节点、历史案例——这正是 px-agent.html 里"依据链追溯"板块要的东西。
  • 反事实(what-if 某装置不检修)由图引擎算,LLM 只组织语言——LLM 自身反事实推理是公认弱项(CounterBench)。

🎓 概念详解:QPN / PyReason / KG as Code

这一页把报告里反复出现的三个核心概念讲透,并对应到已经跑通的 MVP 代码(kg-mvp/ 目录),每个概念都能落到具体文件。

1. QPN(定性概率网络)—— 我们推理器的理论基础

QPN = Qualitative Probabilistic Network,Wellman 1990 年提出,可以理解为"不填数字的贝叶斯网络"。贝叶斯网络要求每个节点给条件概率表(比如"检修导致供给减少 30% 的概率是 0.7"),而研究员根本给不出这种数字;QPN 退一步:只要求方向——边上标 +(同向)或 −(反向),这恰好就是研究员天然会说的"利多/利空"。

推理 = 符号传播(sign propagation),只有两条运算规则:

规则① 串联相乘(⊗):沿路径传
干旱↑ ─(−)→ 单产↓ ─(+)→ 产量↓ ─(+)→ 库存↓ ─(−)→ 价格↑
符号一路相乘:(−)×(+)×(+)×(−)= +,所以干旱对价格是利多。负负得正,和乘法一样。
规则② 并联相加(⊕):多条路径汇聚
两条利多路径汇到同一节点:+ ⊕ + = +(更确定的利多)。
一条利多一条利空:+ ⊕ − = ?(不确定)——这是 QPN 的著名短板,叫 trade-off 问题。

trade-off 问题我们怎么处理:学术方案是给影响分强弱等级让强者胜出;我们的选择更适合交易台——不裁决,把利多路径列表和利空路径列表都摆出来(各自带强度分),因为"多空冲突"本身就是最有价值的研究信号。棕榈油 MVP 里"MPOB 库存超预期对进口利润的影响"就输出了 0.73 vs 0.73 的完美对冲,这个结论是对的:外盘跌降低进口成本(利好利润)但内盘跟跌又压利润,净效应取决于内外盘跌幅差——图谱把这个结构暴露出来,数值交给人查。

落地对应:kg-mvp/propagate.py(约 150 行)就是 QPN 符号传播的完整实现——DFS 遍历 + 符号相乘 + 强度相乘(取代严格的 ⊕,弱边自然衰减)+ 时滞累计 + 环路处理(路径内不重访;回到源点单独记为反馈环,并按首尾符号判定正反馈/负反馈)。

2. PyReason —— 现成的"图上规则推理"开源库

PyReason 是亚利桑那州立大学实验室的开源 Python 库(BSD 协议,pip install pyreason 即装),做的事情一句话:在一张图上,按你写的逻辑规则,一个时间步一个时间步地推出新结论。

输入① 一张图(直接吃 networkx/GraphML,节点和边可带属性)② 一组规则,形如:价格利多(Y) ← 供给减少(X), 边(X→Y), Δt=2(若 X 供给减少且 X 指向 Y,则 2 个时间步后 Y 价格利多)
注解区间每个事实带置信区间,如 供给减少(PX):[0.7, 1.0]——"至少七成把握"。规则传导时区间跟着收紧/放宽,天然表达了"利多但不确定多强"
时间步规则可声明"Δt 步后生效"——检修 2 周后影响现货、6 周后影响库存,这类时滞语义直接写进规则;反馈环在时间展开后自然消解
输出每个时间步每个节点的结论区间 + 完整解释轨迹(哪条规则、哪些前提推出的,可导出 CSV)——可解释性正是交易场景刚需
 
import pyreason as pr
pr.load_graphml('palm.graphml')                # ① 图
pr.add_rule(pr.Rule(
  'price_up(Y) <-1 supply_down(X), edge(X,Y)', # ② 规则:1 步后传导
  'transmission'))
pr.add_fact(pr.Fact('supply_down(印尼出口)', [0.8,1], 0, 5))  # ③ 冲击(带区间、起止时间)
interp = pr.reason(timesteps=6)                # ④ 推理 6 个时间步
# 输出:每步每节点的区间 + 规则触发轨迹

和我们自写传播器的关系:MVP 阶段自写(200 行内)是为了完全掌控冲突消解和输出格式,验证"我们到底需要什么语义";两者是同一范式(图+规则+前向推理)。升级到 PyReason 的触发条件:需要多时间步演化模拟(比如"禁令后第 1/2/3 个月分别发生什么"的逐月推演)、或规则里大量用到置信区间时。YAML 边表不用改,写个转换器即可。

3. KG as Code —— 知识图谱的工程组织方式

类比软件行业的 Infrastructure as Code(用 Terraform 代码定义服务器,而不是手工在控制台点):KG as Code = 用 git 里的文本文件定义图谱,图数据库只是"编译产物"。不是自创概念——LinkML(生物医学界的大型 KG 都这么建,2025 年有 GigaScience 论文)是这条路线最成熟的框架。

git 仓库(唯一权威·事实源)          CI / 本地构建              运行时(随时可重建/可替换)
┌──────────────────────┐    ┌─────────────────────┐    ┌──────────────────────────┐
│ markdown 知识库(给人读) │    │ build_graph.py       │    │ networkx 内存图           │
│ YAML 边表(给机器读)   ──┼──→│  · schema 校验        │──→│  + propagate.py 推理      │
│  polarity/lag/strength │    │  · 悬空节点/非法枚举   │    │  + gen_panel.py 监测面板  │
│  condition/evidence    │    │  · 反馈环检测         │    │  →(未来)MCP tools 给 agent│
└──────────────────────┘    └─────────────────────┘    └──────────────────────────┘
好处具体含义
知识变更走 PR review研究员改一条边的方向/强度,提交 PR,diff 里清清楚楚一行 YAML;错了 git revert 回滚
与现有知识库同一工作流markdown 讲机制(人读)+ 伴生 YAML 存边表(机器读),同一个 PR 一起提交,不存在"两套维护"
存储引擎零锁定图是纯函数构建的产物:今天 networkx,明天要可视化就加灌一份 Neo4j,要 SPARQL 就转 RDF——边表一个字不用改
可测试"干旱必须传导为利多价格"可以写成单元测试,CI 里跑,防止边表改坏

已落地实例:kg-mvp/ 整个目录就是一个 KG-as-code 项目——data/palm_oil.yaml(事实源:43 节点/53 边/23 个带监测信号的根因)→ build_graph.py(校验+构建)→ propagate.py(推理)→ ask_palm.py / gen_panel.py(消费)。从零到跑通一个品种约一个下午,其中大部分时间花在整理边表内容上——这正说明资产是知识本身,代码只是薄薄一层。

4. 名词速查表(本报告出现过的其他术语)

术语一句话解释我们用不用
FCM 模糊认知图QPN 的近亲:边权重是 [-1,1] 连续值且允许环路;研究员脑中的供需框架本质就是一张 FCM✔ 我们的强度权重(强0.9/中0.6/弱0.3)就是 FCM 式的
Datalog一种只有"若…则…"规则的逻辑编程语言,擅长递归推导(A→B→C 自动传导闭包)范式对口但现成引擎要么商业(RDFox)要么集成差,规模小到不需要
OWL / RDF / 本体W3C 语义网标准:RDF 用三元组存知识,OWL 定义概念层级并可推理"分类"问题(X 是一种 Y)✘ 强项是分类学不是业务因果,Java 生态,错配
SPARQL / Cypher图查询语言(分别对应 RDF 库 / Neo4j),类比图数据库上的 SQL暂不用;需要 agent 自由探索图时再上 Neo4j+Cypher
GraphRAG让 LLM 从海量文档自动抽实体关系建图、再按图检索回答问题的一套流程✘ 不走其建图管线(图是人工建的);只借"路径检索"思想
Neuro-symbolic 神经符号混合LLM(神经)管理解和表达,规则引擎(符号)管逻辑推导,各干擅长的事✔ 目标架构:LLM 管两端(新闻→根因、路径→观点),符号引擎管中间传导
MCPModel Context Protocol,让 LLM agent 以标准方式调用外部工具的协议✔ 下一步把 propagate()/监测面板包成 MCP tools 喂给 agent
Reification / RDF-star"把一条边本身当成节点"的建模技巧,用来给边再挂属性(如"某 regime 削弱这条边")备用;目前 condition 字段够用
🎯 三个概念在项目里的分工一句话:KG as Code 解决"知识放哪、怎么协作"(YAML+git);QPN 解决"怎么在图上推利多利空"(符号传播,已自写实现);PyReason 是将来需要更强时序推理时的现成升级件。最新进展已用棕榈油(DCE P)完整落地,见 palm-mvp.html。
