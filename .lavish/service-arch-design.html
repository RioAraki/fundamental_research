先回答你圈的两个概念:networkx 和 vis-network

🧮 networkx(后端"算图"的库)
Python 的图数据结构库。作用:把 YAML 里的"节点+边"在内存里装成一张有向图对象, 让推理引擎能沿边遍历、找反馈环、算可达性、跑符号传播。
它不是数据库、也不是画图工具,就是"程序里表示并计算图"的工具—— 类比 list/dict,只不过数据结构是"图"。我们现在的 build_graph.py 已经在用它。
🎨 vis-network(前端"画图"的库)
浏览器里的 JavaScript 图可视化库。把"节点+边"渲染成可拖拽、缩放、点击的 力导向关系图——就是你在"交互探索"页看到的那种彩色气泡关系图。
我们现在的 lavish 交互探索页已经在用它,只是数据是"烤"进静态 HTML 的。
两者关系:同一份"节点+边"数据,networkx 在后端管逻辑与计算,vis-network 在前端管画面与交互。 后端把 networkx 的图转成 JSON,通过 API 发给前端,vis-network 负责画出来并响应你的点击。v1 要做的核心就是把这条"后端算→前端画"的链路从静态 HTML 变成实时服务。

v1 架构(极简)

整条链路:浏览器 → React 前端(vis-network 画图)→ REST API → FastAPI 后端(复用现有 Python 引擎)→ 读本地 YAML。 没有数据库、没有调度器、没有反向代理、没有部署层。数据就用现在 data/copper.yaml + copper_events.yaml(已经是"make up 的数据")。

v1 具体做什么

后端端点(FastAPI)复用前端(React + vis-network)
GET /api/graphbuild_graph + gen_explore 逻辑画出整张沪铜关系图(节点着色/边红蓝)
GET /api/node/{id}gen_explore 的节点解释点节点 → 侧栏显示含义/方向/利好利空判定/信号/主题
GET /api/edge/{id}边的五要素 + rationale点边 → 显示相互作用与"为什么这个强度"
POST /api/reasonpropagate情景推演:勾选冲击(或选某日活跃事件)→ 高亮传导路径 + 分时段结论

就这 4 个端点 + 一个交互图谱页面。本地起法:后端一条 uvicorn 命令、前端一条 npm run dev 命令(或前端构建后由后端直接托管静态文件,单进程起)。要跑在服务器上时同样两条命令,不需要额外部署设施。
